# ERC-8004 for GOAT Network

ERC-8004 defines a trust layer for agent ecosystems using three registries:

- **Identity Registry** (agent discovery and ownership)
- **Reputation Registry** (feedback and scoring signals)
- **Validation Registry** (third-party verification hooks)

This page is an implementation-focused guide for GOAT builders.

## Standard Status (Important)

<Callout type="info">
  ERC-8004 is publicly published in the Ethereum ERC process and has live deployments across multiple EVM networks.
  In the canonical ERC repository, the current status flag is still <strong>Draft</strong> (not Final), so builders should track spec and contract updates.
</Callout>

## How to Use ERC-8004 (Builder Flow)

### 1) Select registry addresses for your target chain

Use the official deployment list, then build:

- `agentRegistry = eip155:{chainId}:{identityRegistryAddress}`

Examples:

- `eip155:2345:0x8004A169FB4a3325136EB29fA0ceB6D2e539a432`
- `eip155:48816:{confirmed-testnet-identity-address}`

### 2) Register an agent (Identity Registry)

Core operations:

- `register(agentURI)`
- `setAgentURI(agentId, newURI)`
- optional metadata and wallet methods depending on implementation version

`agentURI` should resolve to a registration JSON file (IPFS or HTTPS recommended).

### 3) Submit reputation signals (Reputation Registry)

Core operation:

- `giveFeedback(agentId, value, valueDecimals, tag1, tag2, endpoint, feedbackURI, feedbackHash)`

Read paths typically include:

- `readFeedback(...)`
- `readAllFeedback(...)`
- `getSummary(...)`

### 4) Add validation proofs (Validation Registry, where deployed)

Typical workflow:

- `validationRequest(...)`
- `validationResponse(...)`

### 5) Combine with payment rails

ERC-8004 is a trust and discovery layer. Payment and settlement (for example x402-style flows) are complementary and should be integrated as a separate layer.

## Operational Recommendations

1. **Treat registration as identity, not automatic trust.**
2. **Use anti-Sybil logic** for production reputation views.
3. **Store off-chain evidence with URI + hash** for integrity.
4. **Use stronger validation models** as value-at-risk increases.
5. **Pin implementation to audited contract releases** and monitor ERC updates.
