# ERC-8004 for GOAT Network

ERC-8004 defines a trust layer for agent ecosystems using three registries:

- **Identity Registry** for agent discovery and ownership
- **Reputation Registry** for feedback and scoring signals
- **Validation Registry** for third-party verification workflows

This page focuses on how GOAT builders should evaluate and integrate the standard today.

## Standard Status

<Callout type="info">
  ERC-8004 is publicly published in the Ethereum ERC process, but the canonical ERC markdown still marks it as <strong>Draft</strong>, not Final. Treat current contracts and docs as implementation-ready but still evolving.
</Callout>

## Builder Flow

### 1) Choose the registry for your target chain

Use the Identity Registry address for your chain and construct the `agentRegistry` identifier as:

- `eip155:{chainId}:{identityRegistryAddress}`

Example on GOAT:

- `eip155:2345:0x8004A169FB4a3325136EB29fA0ceB6D2e539a432`

### 2) Register the agent in Identity Registry

The canonical implementation supports three registration entry points:

- `register()`
- `register(agentURI)`
- `register(agentURI, metadata)`

Useful follow-up methods include:

- `setAgentURI(agentId, newURI)`
- `getMetadata(agentId, metadataKey)`
- `setMetadata(agentId, metadataKey, metadataValue)`
- `setAgentWallet(agentId, newWallet, deadline, signature)`
- `unsetAgentWallet(agentId)`

`agentURI` should resolve to a registration JSON document, typically over IPFS or HTTPS.

<Callout type="info">
  `agentWallet` is a reserved payment-receiving field in the spec. It defaults to the owner wallet on registration and should be changed with `setAgentWallet(...)`, not by writing generic metadata.
</Callout>

### 3) Publish a registration JSON that matches the spec

In the current ERC-8004 spec, the following top-level fields are expected in the registration document:

- `type`
- `name`
- `description`
- `image`
- `services`
- `x402Support`
- `active`
- `registrations`

`supportedTrust` is optional.

### 4) Collect trust signals through Reputation Registry

Primary write path:

- `giveFeedback(agentId, value, valueDecimals, tag1, tag2, endpoint, feedbackURI, feedbackHash)`

Common read paths:

- `readFeedback(...)`
- `readAllFeedback(...)`
- `getSummary(...)`

For production integrations, keep the on-chain record compact and store richer evidence off-chain behind `feedbackURI`, with `feedbackHash` when you need integrity verification.

### 5) Use Validation Registry only when the deployment is confirmed

The canonical contracts expose:

- `validationRequest(validatorAddress, agentId, requestURI, requestHash)`
- `validationResponse(requestHash, response, responseURI, responseHash, tag)`

That interface is usable, but the surrounding Validation Registry section is still evolving. Confirm the address, trust model, and validator process on your target chain before shipping.

### 6) Combine ERC-8004 with payment rails when needed

ERC-8004 is a trust and discovery layer. If your agent also charges for access, combine it with a payment protocol such as x402 rather than treating ERC-8004 itself as the payment layer.

## Registration JSON Example

```json
{
  "type": "https://eips.ethereum.org/EIPS/eip-8004#registration-v1",
  "name": "GoatAgent",
  "description": "Autonomous service agent on GOAT Network",
  "image": "https://example.com/agent.png",
  "services": [
    {
      "name": "A2A",
      "endpoint": "https://agent.example/.well-known/agent-card.json",
      "version": "0.3.0"
    },
    {
      "name": "MCP",
      "endpoint": "https://mcp.agent.example/",
      "version": "2025-06-18"
    }
  ],
  "x402Support": true,
  "active": true,
  "registrations": [
    {
      "agentRegistry": "eip155:2345:0x8004A169FB4a3325136EB29fA0ceB6D2e539a432",
      "agentId": 1
    }
  ],
  "supportedTrust": ["reputation", "crypto-economic", "tee-attestation"]
}
```

If you are using ERC-8004 only for discovery, you can omit `supportedTrust` or leave it empty. Set `x402Support` to reflect your real payment capabilities rather than treating it as a decorative field.

## Operational Recommendations

1. **Treat registration as discoverability, not automatic trust.**
2. **Pin the registration JSON and any feedback evidence** to stable storage such as IPFS when possible.
3. **Use `feedbackURI` + `feedbackHash`** for auditable off-chain evidence.
4. **Set a verified agent receiving wallet** with `setAgentWallet(...)` when payments should not go to the owner address.
5. **Track spec and contract updates** because ERC-8004 is still Draft.
